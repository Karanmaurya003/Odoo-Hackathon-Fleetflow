Relationships Explained
REL — 01 / 04
Vehicles ↔ Trips 1 : N
VEHICLES ──────▷ TRIPS
A single vehicle can be assigned to many trips over its lifetime — sequentially, never concurrently. Each trip record holds exactly one vehicle_id foreign key, pointing back to the parent vehicle.
vehicles.id (PK) ──1──▷──N── trips.vehicle_id (FK)
ON DELETE RESTRICT | NOT NULL
Rule: vehicle.status = 'Available' to create trip
Rule: vehicle.status → 'OnTrip' when dispatched
REL — 02 / 04
Drivers ↔ Trips 1 : N
DRIVERS ──────▷ TRIPS
One driver can complete many trips across their career. Each trip is assigned to exactly one driver via driver_id. The system enforces license validity and duty status before any assignment.
drivers.id (PK) ──1──▷──N── trips.driver_id (FK)
ON DELETE RESTRICT | NOT NULL
Rule: driver.status = 'OffDuty' to be assigned
Rule: driver.license_expiry > TODAY() required
REL — 03 / 04
Vehicles ↔ Fuel Logs 1 : N
VEHICLES ──────▷ FUEL_LOGS
Every refuelling event is logged against a specific vehicle. One vehicle accumulates many fuel records over time. These records power the cost-per-km and fuel efficiency analytics calculations.
vehicles.id (PK) ──1──▷──N── fuel_logs.vehicle_id (FK)
ON DELETE RESTRICT | NOT NULL
Analytics: fuel_efficiency = Σkm / Σliters
Analytics: cost_per_km uses Σfuel_cost + Σmaint_cost
REL — 04 / 04
Vehicles ↔ Maintenance 1 : N
VEHICLES ──────▷ MAINTENANCE
A vehicle can have many maintenance records — oil changes, tyre replacements, inspections. Creating any maintenance record automatically locks the vehicle with status 'InShop', removing it from dispatch selection.
vehicles.id (PK) ──1──▷──N── maintenance.vehicle_id (FK)
ON DELETE RESTRICT | NOT NULL
Side-effect: vehicle.status → 'InShop' on INSERT
Side-effect: vehicle.status → 'Available' on complete

Integrity Constraints & Business Rules
Vehicle Constraints
  • license_plate is globally UNIQUE
  • status ∈ {Available, OnTrip, InShop, Retired}
  • Only status=Available vehicles are dispatch-eligible
  • status=Retired vehicles cannot be deleted if trips exist
  • odometer is updated to end_odometer on trip completion
Driver Constraints
  • status ∈ {OffDuty, OnDuty, Suspended}
  • Only status=OffDuty drivers are dispatch-eligible
  • license_expiry must be > CURDATE() to be assigned
  • Suspended drivers cannot be assigned to any trip
  • safety_score tracked per driver (0–100 scale)
Trip Constraints
  • status ∈ {Draft, Dispatched, Completed, Cancelled}
  • cargo_weight must be ≤ vehicle.max_capacity
  • end_odometer must be > start_odometer
  • Both vehicle_id and driver_id are NOT NULL
  • revenue is summed per vehicle for ROI analytics
Referential Integrity
  • All FK references use ON DELETE RESTRICT
  • Vehicles with trips cannot be hard-deleted
  • Drivers with trips cannot be hard-deleted
  • email in users table is UNIQUE indexed
  • All monetary values stored as FLOAT (precision aware)
FuelLog Rules
  • vehicle_id FK is NOT NULL — orphan logs disallowed
  • liters and cost must be > 0 (application-level)
  • date defaults to CURDATE() if not supplied
  • Aggregated into monthly summaries for reporting
  • Used in: fuel_efficiency, cost_per_km, ROI
Maintenance Rules
  • INSERT auto-sets vehicle.status = 'InShop'
  • Cannot add maintenance to OnTrip vehicles
  • Cannot add maintenance to Retired vehicles
  • Complete action checks for other pending records first
  • cost aggregated into operational cost analytics
